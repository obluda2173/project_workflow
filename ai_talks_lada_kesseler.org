* Coding with AI — Patterns (Llewellyn Kesler)

** Core constraint: LLMs can't learn from you
- Weights are frozen, model is stateless
- "Memory" in chat UIs is just context window concatenation
- You can only *append* to context or *reset* it

** Context degrades
- Performance isn't uniform — starts good, weakens, then plummets
- Happens well before hitting the token limit
- Active context management is non-optional

** Pattern 1 — Knowledge Documents
Extract valuable discoveries into files /before/ resetting context.
On reset: load file → clean context + retained knowledge.
Stops you from being a "human copy-paste machine."

** Pattern 2 — Ground Rules
Files *always* loaded (=.claude/rules=, Windsurf rules).
Only the most critical stuff: "prefer simple solutions," "warn me before I do something dumb."
User-level and project-level.

** Pattern 3 — Extract Knowledge
Dirty context + good gems → ask agent to save to file.
Same for modifying existing knowledge docs.

** Anti-pattern — Distracted Agent
One agent doing everything = diluted attention.
Result: shallow focus or fixation on irrelevant details.
Instructions like "warn me about bad practices" silently stop working. Examples:
- Be very honest. Tell me something I need to know even if I don't want to hear it.
- Be proactive and flag issues before they become problems.

** Pattern 4 — Focused Agents
Specialist agents in parallel. Example: dedicated "committer" agent.
Same ground rules, but focused context → actually catches problems.
Small focused agent >> large scattered agent.

** Pattern 5 — Reference Docs
Build a library of on-demand docs (style guides, test plans, scratch pads).
Pull in as needed — granular control over context occupation.

** Pattern 6 — Knowledge Composition
Split reference docs by concern.
Same logic as not writing giant functions: one file = one pull-in = better focus.

** Pattern 7 — Semantic Zoom
Text is elastic. Zoom out: "TLDR / high-level." Zoom in: "tell me more."
Use cases:
- *New codebases* — high-level architecture, ASCII diagrams
- *Debugging blind* — high-level explanation → spot nonsense fast → peel layers
- *Tests* — approval tests formatted for at-a-glance scanning
- *Doc maintenance* — keep succinct, delete mercilessly, temporary vs permanent

** Pattern 8 — Noise Cancellation
LLMs are verbose by default. Put "be succinct" in ground rules.
Match detail level to curiosity level.
One noisy AI is bad; four devs × noisy AI = chaos.

** Pattern 11 — Knowledge Checkpoint
Before telling the agent to implement, save the plan to a file and commit.
If implementation fails, rollback costs minutes, not the 30min you spent explaining.
Protects your most valuable asset: time spent talking to it.
Fight the default — Claude will try to implement immediately ("Labrador puppy"). Interrupt it: "no, save first."

** Pattern 12 — Parallel Implementations
Roll of a dice → roll five dice.
From a checkpoint, fork into multiple branches (=git worktree=), open separate agents, let them run in parallel.
- Trade tokens for your waiting time
- Pick the best result, or cherry-pick across implementations ("I like this layout + those buttons")
- Combine outputs into one solution better than any single attempt

** Pattern 13 — Offload Determinism
Don't use the agent for repeatable deterministic tasks.
Example: instead of "take a screenshot via puppeteer MCP" (unreliable), have it *write a script* that does it.
Now you own a deterministic tool. Agent can still call it, but the brittle part is gone.

** Anti-pattern 14 — Perfect Recall Fallacy
Trained on the internet ≠ remembers the internet. Don't expect library-level detail.
Code is self-verifiable — run it. If you trust output without running, that's on you.

** Pattern 15 — Playgrounds
When stuck debugging from UI level, pull back. Have the agent write isolated dirty code to explore the problem.
Example: chess library silently required a king on the board — unit test surfaced it instantly.
Add a ground rule: "when stuck, experiment in a scratch directory ("playground" in gitignored)."
Extract findings into reference docs → build a library of verified knowledge.

** Pattern 16 — Invalidated Leaps
AI makes unvalidated assumptions mid-implementation — same bug as a human coding 3 hours without running.
When stuck: slow it down, tell it to validate each assumption.
*Predictive TDD* works especially well here:
- Predict outcome → run → surprise = broken mental model
- Works on the agent too — snaps it out of loops
- Extract your TDD process into a reference doc, reuse everywhere

** Pattern 17 — Chain of Small Steps
AI fails under complexity. You manage the decomposition, not the agent.
Example (version number display):
1. Hardcode a version variable, display it → trivial
2. Write a script that increments that variable → trivial
3. Wire it into a git hook → trivial
Each step: focused agent, one concern, knowledge checkpoint before next step.
Combines: chain of small steps + offload determinism + focused agent + knowledge checkpoint.

** Tricky instructions — fighting agent training
Some behaviors are baked into weights (e.g. code comments, Whisper hallucinating "like and subscribe" on silence).
You cannot prompt these out. Filter or force them out instead.

** Pattern 18 — Hooks (Claude Code)
Deterministic injection points in the agentic loop.
- Hook into =first_message=, =edit=, =conversation_start=, etc.
- Example: Python hook on =edit= event detects new comments via tree-sitter → screams at agent → agent removes them
- Reliable where prompting fails

** Pattern 19 — Exploit Recency Bias
The agent values recent instructions more. Exploit this:
- *To-do lists* — written checklist >> verbal dump. Focuses the agent.
- *Instruction sandwich* — wrap implementation steps with what you care about (e.g. "run tests" before and after).
- *Reminder hooks* — automate nagging. Example: hook that prepends your top 5 rules to every prompt. Only method that reliably enforces "ask questions one at a time."
** Anti-pattern 20 — Silent Misalignment
AI has a hidden "mental model" + compliance bias = silently does the wrong thing without asking.
Example: told it to level two UI panels. It agreed, failed repeatedly. Asked it to color what it thought the "header" was — completely different DOM understanding than yours.
Never said "I don't understand what you mean." Just kept trying.

** Pattern 21 — Active Partner
Suppress default compliance. Ground rules:
- Tell me something I need to know even if I don't want to hear it.
- Push back when something seems wrong. Don't agree with mistakes.
- Ask questions if something's not clear, especially before making a choice.
- Show a specific emoji (e.g. ⚠️) when flagging a potential error — confirms the rule is active.
Reinforce with prompts: "what do you really think?", "do you have any questions?", "is everything clear?"
Don't say "give me 3 questions" — arbitrary caps suppress what might be the most important question, or force invented ones.

** Pattern 22 — Check Alignment
Before implementation: "show me what you're about to do."
Two benefits: catches misunderstandings early + seeing a plan triggers your own opinions about what to change.
Cheaper to correct a plan than to undo wrong implementation.

** Pattern 23 — Context Markers
Start replies with a starter character (emoji). Confirms rules are loaded.
- Disappears → context is degraded, time to reset.
- Stack emojis per specialist agent (committer gets its own).
- Process files get their own marker — confirms the agent is following the process, not just winging it.
- Can attach to individual instructions in ad-hoc prompts to verify they're being followed.
Loop-to-loop: tell it at end of a big chunk to reread core rules. Costs context, reinforces rules. Still undecided on tradeoff.

** Anti-pattern 24 — Answer Injection
Your question smuggles in an assumed answer shape, cutting off AI's breadth.
"Which fancy restaurant should I book?" → gets restaurant. Never told you a picnic would be better.
"Can you send a message to another Claude Code?" → researched inter-process comms, said no. But the actual situation (two tmux panes) had a trivial AppleScript solution it never offered — because the question pre-narrowed the search space.
Fix: describe *what you're trying to do*, not *how you think it should be done*.

** Pattern 25 — Reverse Direction
Stop treating the agent as an order taker. Instead of dictating, ask it to suggest.
"Rename this file" → "suggest names for this file."
You see options, recognize what you like, get ideas you wouldn't have had.
Leverages AI's breadth — the strongest thing it has. Augments you instead of reducing you to a dispatcher.

** Pattern 26 — Text Native
Text is AI's strongest modality. Stay in text as long as possible.
ASCII as lightweight visual medium — like markdown is to code, ASCII is to images.
Example workflow (UI redesign at hackathon):
1. Feed hand-drawn mockup images → ask AI to convert to ASCII
2. Edit the ASCII (lightweight whiteboard you both share)
3. Ask AI to describe the ASCII in prose → now you have a spec
4. Feed current code state → ask AI to describe it the same way
5. Diff the two descriptions → to-do list writes itself
6. Implement from the to-do list → worked first try
Took a giant leap by staying in text and letting AI bridge representations.

** Takeaways (speaker's summary)
- LLMs can't learn → you build the memory system (files, reference library)
- Focus is scarce → small agents beat big agents
- Text is elastic → zoom in/out, stay text-native, keep docs scannable
- Non-determinism → protect (checkpoints) or exploit (parallel runs)
- Don't expect perfect recall → let it experiment, save verified knowledge
- Tricky instructions → hooks and reminders, not prompting harder
- Mental model is hidden + compliance bias → actively fight silent agreement
- Breadth is AI's strongest asset → stop narrowing with pre-shaped questions
